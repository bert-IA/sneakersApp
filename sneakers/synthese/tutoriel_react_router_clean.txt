PARTIE 1 : COMPRENDRE LES SINGLE PAGE APPLICATIONS
===================================================

1.1 QU'EST-CE QU'UNE SINGLE PAGE APPLICATION ?
-----------------------------------------------

DÉFINITION TECHNIQUE
Une Single Page Application (SPA) est une application web qui charge une seule page HTML initiale, puis met à jour dynamiquement le contenu sans recharger complètement la page.

COMPARAISON : SITES TRADITIONNELS VS SPA

SITES TRADITIONNELS (Multi-page)

Utilisateur clique sur un lien (ex: "Page B")
    ↓
Navigateur envoie une requête HTTP au serveur
    ↓
Serveur génère une nouvelle page HTML complète
(Header, navigation, contenu, footer, CSS, JS)
    ↓
Navigateur reçoit la réponse et recharge ENTIÈREMENT la page
    ↓
Flash blanc pendant le rechargement + perte de l'état JavaScript
    ↓
L'utilisateur voit la nouvelle page (délai de chargement visible)


SINGLE PAGE APPLICATIONS

Utilisateur clique sur un lien (ex: "Page B")
    ↓
JavaScript (React Router) intercepte le clic
    ↓
React Router change l'URL dans la barre d'adresse (/page-b)
    ↓
React Router compare l'URL aux routes définies :
    - Route "/page-a" → ComposantA ? NON
    - Route "/page-b" → ComposantB ? OUI ✓
    ↓
React Router affiche le ComposantB
(Le composant est DÉJÀ chargé en mémoire, pas de requête serveur)
    ↓
Transition immédiate et fluide vers le nouveau contenu
(Seule la zone de contenu change, header/navigation restent)


EXEMPLES CONCRETS DE SPA
- Gmail : Navigation entre boîte de réception, envoyés, corbeille sans rechargement
- Facebook : Scroll infini, navigation entre profils, timeline
- Netflix : Navigation entre catégories, lecteur vidéo
- Spotify Web : Navigation entre playlists, recherche, lecture

AVANTAGES DES SPA

EXPÉRIENCE UTILISATEUR
- Fluidité : Pas de "flash blanc" entre les pages
- Rapidité : Pas de rechargement complet du DOM
- Interactivité : Animations et transitions fluides
- Offline : Possibilité de fonctionner hors-ligne (avec service workers)

PERFORMANCE TECHNIQUE
- Chargement initial : CSS/JS chargés une seule fois
- Données : Seules les données nécessaires sont récupérées (API REST/GraphQL)
- Cache navigateur : Meilleure utilisation du cache
- Bande passante : Moins de transferts de données

DÉVELOPPEMENT
- État de l'application : Plus facile à maintenir
- Composants réutilisables : Architecture modulaire
- Debugging : État centralisé plus simple à déboguer

INCONVÉNIENTS À CONNAÎTRE

COMPLEXITÉ INITIALE
- Bundle size : Application plus lourde au premier chargement
- SEO : Plus complexe pour l'indexation (résolu avec SSR/SSG)
- Navigation : Gestion manuelle de l'historique du navigateur

DÉFIS TECHNIQUES
- Memory leaks : Gestion de la mémoire sur une longue session
- État complexe : Plus de logique côté client
- Accessibilité : Navigation au clavier et lecteurs d'écran

1.2 POURQUOI REACT ROUTER ?
----------------------------

LE PROBLÈME SANS ROUTER
Dans notre application actuelle, tout est dans un seul composant App.jsx :

// Structure actuelle - tout mélangé
function App() {
  return (
    <div>
      <Banner />           // Header toujours visible
      <ShoppingList />     // Catalogue des sneakers
      <Cart />            // Panier toujours visible
    </div>
  );
}

LIMITATIONS DE CETTE APPROCHE
1. EXPÉRIENCE UTILISATEUR MÉDIOCRE
   - Tout est visible en même temps
   - Pas de séparation logique des fonctionnalités
   - Interface encombrée

2. NAVIGATION IMPOSSIBLE
   - Pas d'URL spécifique pour le panier
   - Impossible de partager un lien vers le panier
   - Bouton "Retour" du navigateur ne fonctionne pas

3. MAINTENABILITÉ DIFFICILE
   - Logique de plusieurs pages dans un seul composant
   - État global difficile à gérer
   - Code difficile à organiser

LA SOLUTION REACT ROUTER
React Router résout ces problèmes en :

1. SÉPARATION DES RESPONSABILITÉS
   - Une page = un composant spécialisé
   - URL spécifique pour chaque page
   - Navigation logique entre les sections

2. GESTION AUTOMATIQUE
   - Historique du navigateur (boutons précédent/suivant)
   - URL dans la barre d'adresse
   - Rechargement de page sur la bonne section

3. ARCHITECTURE PROPRE
   - Composants indépendants et réutilisables
   - Code organisé par fonctionnalité
   - Maintenance simplifiée

2.2 CONCEPTS FONDAMENTAUX DE REACT ROUTER
------------------------------------------

LES COMPOSANTS ESSENTIELS

1. BrowserRouter
   - Composant racine qui active le routing
   - Utilise l'API HTML5 History
   - Doit englober toute l'application

2. Routes
   - Container pour définir les routes
   - Remplace l'ancien Switch de React Router v5

3. Route
   - Définit une route spécifique
   - Associe un chemin à un composant
   - Props principales : path et element

4. Link
   - Crée des liens de navigation
   - Remplace les balises <a> traditionnelles
   - Évite le rechargement de page

5. useNavigate
   - Hook pour navigation programmatique
   - Remplace l'ancien useHistory
   - Permet de naviguer depuis le JavaScript

EXEMPLE DE BASE

import { BrowserRouter, Routes, Route, Link } from 'react-router-dom';

function App() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Accueil</Link>
        <Link to="/cart">Panier</Link>
      </nav>
      
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/cart" element={<CartPage />} />
      </Routes>
    </BrowserRouter>
  );
}

EXPLICATION DÉTAILLÉE DE CET EXEMPLE

DIFFÉRENCE FONDAMENTALE : NAVIGATION VS AFFICHAGE

Il faut bien comprendre que nous avons DEUX systèmes qui travaillent ensemble :

1. LES LIENS DE NAVIGATION (ce qu'on clique)
2. LES ROUTES D'AFFICHAGE (ce qu'on voit)

PARTIE NAVIGATION - LES LINK

<Link to="/">Accueil</Link>

Ce que fait cette ligne :
- Crée un lien cliquable avec le texte "Accueil"
- Quand on clique dessus, change l'URL vers "/"
- Remplace un <a href="/"> traditionnel
- N'affiche RIEN par lui-même, c'est juste un bouton

<Link to="/cart">Panier</Link>

Ce que fait cette ligne :
- Crée un lien cliquable avec le texte "Panier"  
- Quand on clique dessus, change l'URL vers "/cart"
- N'affiche RIEN par lui-même, c'est juste un bouton

PARTIE AFFICHAGE - LES ROUTE

<Route path="/" element={<HomePage />} />

Ce que fait cette ligne :
- Surveille l'URL de la page
- SI l'URL est exactement "/", ALORS affiche le composant <HomePage />
- Ne crée AUCUN lien cliquable, c'est purement de l'affichage

<Route path="/cart" element={<CartPage />} />

Ce que fait cette ligne :
- Surveille l'URL de la page
- SI l'URL est exactement "/cart", ALORS affiche le composant <CartPage />
- Ne crée AUCUN lien cliquable, c'est purement de l'affichage

COMMENT ÇA MARCHE ENSEMBLE - LE CYCLE COMPLET

1. L'utilisateur voit deux boutons : "Catalogue" et "Panier"

2. L'utilisateur clique sur "Panier"
   - Le <Link to="/cart"> détecte le clic
   - React Router empêche le comportement par défaut du navigateur
   - L'URL change vers "/cart" (sans requête serveur)

3. React Router détecte le changement d'URL automatiquement
   - Il parcourt toutes les <Route> définies dans <Routes>
   - Il trouve <Route path="/cart" element={<CartPage />} />
   - Il décide d'afficher le composant <CartPage />

4. React re-rend SEULEMENT la zone de contenu
   - Banner reste identique (pas de re-rendu)
   - Seule la zone <Routes> change de contenu
   - CartPage remplace HomePage instantanément

5. L'utilisateur voit maintenant le contenu de CartPage
   - Transition immédiate (0ms)
   - Aucun flash blanc
   - État de l'application conservé (panier, données, formulaires, etc.)

ANALOGIE SIMPLE

Imaginez une télévision :

LES LINK = Les boutons de la télécommande
- Bouton "1" = <Link to="/">
- Bouton "2" = <Link to="/cart">
- Ils permettent de CHANGER de chaîne

LES ROUTE = Les chaînes de télévision  
- Chaîne 1 = <Route path="/" element={<HomePage />} />
- Chaîne 2 = <Route path="/cart" element={<CartPage />} />
- Elles définissent CE QUI S'AFFICHE sur chaque chaîne

Quand vous appuyez sur le bouton "2", la télé change sur la chaîne 2 !

POURQUOI CETTE SÉPARATION ?

1. FLEXIBILITÉ

3. MAINTENABILITÉ
   - Navigation et contenu sont séparés
   - Plus facile de modifier l'un sans casser l'autre

PROBLÉMATIQUE THÉORIQUE
Avec le routing, nous pourrions rencontrer un défi technique majeur : l'état doit survivre aux changements de composants.

EXEMPLE DE CE QUI POURRAIT MAL SE PASSER (Architecture problématique)

MAUVAISE APPROCHE - État dans les pages individuelles :

HomePage.jsx
├── useState([]) // État du panier ici - ERREUR !
└── ShoppingList

CartPage.jsx  
├── useState([]) // Autre état panier, différent - ERREUR !
└── Cart

SCÉNARIO PROBLÉMATIQUE :

1. UTILISATEUR SUR HOMEPAGE (URL: /)
   - Composants actifs : App → Banner → HomePage → ShoppingList
   - État local dans HomePage : liste des sneakers affichés
   - L'utilisateur ajoute 3 sneakers au panier

2. UTILISATEUR NAVIGUE VERS CARTPAGE (URL: /cart)
   - React Router DÉMONTE HomePage et ShoppingList
   - React Router MONTE CartPage et Cart
   - PROBLÈME : Si l'état du panier était dans HomePage, il est PERDU !
   - PROBLÈME : Si l'état des filtres était dans ShoppingList, il est PERDU !

3. UTILISATEUR RETOURNE À HOMEPAGE (URL: /)
   - React Router DÉMONTE CartPage et Cart
   - React Router MONTE à nouveau HomePage et ShoppingList (NOUVEAUX composants)
   - PROBLÈME : HomePage redémarre à zéro, comme au premier chargement
   - RÉSULTAT : Panier vide, filtres perdus, sélections effacées !

    const addToCart = (sneaker) => {
        setCart(prevCart => {
            const existingItem = prevCart.find(item => item.id === sneaker.id);
            if (existingItem) {
                return prevCart.map(item =>
                    item.id === sneaker.id
                        ? { ...item, quantity: item.quantity + 1 }
                        : item
                );
            } else {
                return [...prevCart, { ...sneaker, quantity: 1 }];
            }
        });
    };

    const removeFromCart = (sneaker) => {

const CartContext = createContext();

export function CartProvider({ children }) {
    const [cart, setCart] = useState(() => {
        const savedCart = localStorage.getItem('cart');
        return savedCart ? JSON.parse(savedCart) : [];
    });

    useEffect(() => {
        localStorage.setItem('cart', JSON.stringify(cart));
    }, [cart]);

    // Calcul automatique du compteur
    const cartItemsCount = cart.reduce((total, item) => total + item.quantity, 0);

    const addToCart = (sneaker) => {
        setCart(prevCart => {
            const existingItem = prevCart.find(item => item.id === sneaker.id);
            if (existingItem) {
                return prevCart.map(item =>
                    item.id === sneaker.id
                        ? { ...item, quantity: item.quantity + 1 }
                        : item
                );
            } else {
                return [...prevCart, { ...sneaker, quantity: 1 }];
            }
        });
    };

    const removeFromCart = (sneaker) => {
        setCart(prevCart => {
            const existingItem = prevCart.find(item => item.id === sneaker.id);
            if (existingItem) {
                if (existingItem.quantity === 1) {
                    return prevCart.filter(item => item.id !== sneaker.id);
                } else {
                    return prevCart.map(item =>
                        item.id === sneaker.id
                            ? { ...item, quantity: item.quantity - 1 }
                            : item
                    );
                }
            }
            return prevCart;
        });
    };

    const clearCart = () => {
        setCart([]);
    };

    return (
        <CartContext.Provider value={{
            cart,
            cartItemsCount,
            addToCart,
            removeFromCart,
            clearCart
        }}>
            {children}
        </CartContext.Provider>
    );
}

export function useCart() {
    const context = useContext(CartContext);
    if (!context) {
        throw new Error('useCart must be used within a CartProvider');
    }
    return context;
}
